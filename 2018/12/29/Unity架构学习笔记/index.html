Unity架构学习笔记

发表于 2018-12-29 | 分类于 Unity ， 架构设计

简介

就像房子需要搭架子，有了架构代码结构也会稳固的多，搭好了架构项目事半功倍，也很容易复用。不然每个代码都一个个monobehavior的写，速度永远提高不上来。

对应视频教程：UNITE －Unity项目架构设计与开发管理

主流架构

  • EmptyGO
  • Simple GameManager
  • Manager of Managers
  • MVCS(StrangeIOC)
  • MVVM（uFrame）
  • ……

EmptyGO

  • 把所有代码挂在空物体上。
  • 用GameObject.Find()或者在Inspector属性面板上（public属性）直接关联彼此

问题：一旦引用关系发生变化，项目规模变大的时候，会造成混乱

Simple GameManager

  • 把管理类改成Singleton
  • 这是最简单也是最常用的Unity架构一种方法。
  • 当其他GameObject想访问这个逻辑的时候只要调getInstance，在任何地方都可以访问
    优点：对于小型项目来说非常方便快捷。很实用
    缺点：不容易扩展

Manager of Managers

多个管理类，每个管理类都有分工

  • 主管理类管理所有的自管理类
  • 子管理类都是单例模式，方便相互调用
  • 这是最基本最主流的方法，适合中型以上的游戏。
  • 这些要和GUI事件严格剥离开
  • 项目之间复用很方便

    Sub-Managers

  • 一个中型以上的项目必须有：
    • Level Manager(场景管理)
    • Pool Manager(池管理)
    • Save Manager(储存管理)
  • 最好有:
    • Game Manager
    • Event Manager
    • Menu Manager
    • Audio Manager

      LevelManager

      场景管理器

      为什么一定需要Level Manager

      Unity默认载入产经函数：Application.LoadLevel(“levelName”),Application.LoadLevel(0)
      如果1~2个level没有关系，多了就找不到了，有可能有场景复用，改变几个参数。
      也有可能要临时改变载入顺序

      A Simple Level Manager Design

  • 写配置表
  • 创建新的Api: LevelManger.LoadNext();
  • 配置表里要要允许给每一关设置一些参数
  • 这些参数要很容易读取调用
  • 允许创建多个配置表，在启动应用的时候切换配置表
  • 这样就很容易能创建不同版本的游戏
  • 插件推荐：MadManager

    衍生

    在切换两个场景之间可以加入不透明度变化、颜色变化、播放一小段动画等等

    Pool Manager

    频繁创建角色、npc。假如每次创建都直接调用，不需要的时候直接销毁，就会频繁的调用C#中的GC.GC是一种非常耗时的操作，因此不推荐。

    A Simple Pool Design

  • 建一个池把临时物体管理起来备用，在需要的时候拿出来。
  • 这个列表包括各种各样不同的GameObject,prefab等
  • 这个池设置一个容量，超出池的容量以后就把以前的删除掉。（先进先出）
    但是这样的设计还是有很多问题。unity是用prefab来管理的。需要先把prefab创建出来。除了临时创建物体，当前活动物体也需要管理。要针对每一种prefab进行管理

    BetterDesign

    推荐的池层级：
  • PoolManager
    • SpawnPool
      • PrefabPool
        ​ √ Active instances.
        ​ √ Inactive instances.
        PoolManager Design
  • PoolManager设计原则
    • 设计成单例
    • 管理Spawn Pools
      SpawnPool Design
  • SpawnPool设计原则
    • 选择一个空物体作为生成池物体们的父物体
    • 用字典管理PrefabPools
      PrefabPool Design
  • 为每一个prefab建立PrefabPool
  • 建立一个可见列表和一个不可见列表
  • 管理prefab加载和卸载的过程
    例：超出一定容量自动销毁或停止创建。但对于npc不能这样设计。确认这个npc没有用才能删除，每次删除需要限定个数

    Save Manager

    玩到某一个节点退出了，再进入想回复进度
    插件推荐：SaveManager(采用二进式，储存加载快)

MVCS:StrangeIOC

StrangeIOC主要用于处理UI events,当ui之间关系变得复杂时，只使用Manager管理就会变得非常混乱，但凡ui发生改变一点东西，其他代码就要大量改变。StrangeIOC会把UI和逻辑分开。否则会对项目进度产生灾难性的影响。StrangeIOC是免费开源的，是一个相对比较好的选择。只是在非UI处理的模块和第三方插件兼容，比如VR、AR的应用上不够直观。

Types Of Binding

KeyValueNotes

interfaceImplementationbinds an interface to its implementation.

Mediator

  • Mediator使UI界面和操作分离开来，修改UI不再需要修改逻辑层面，只需要修改Mediator
  • Binding
    • mediationBinder.Bind<ExampleView>().To<ExampleViewMediator>();

      Dispatcher

  • Simple format.(格式简单）
    dispatcher.Dispatch(Event.FIRE_MISSILE);
  • Event+Data(事件可以包含数据）
    Vector3 orientation=gameObject.transform.localRotation.eulerAngles;
    dispatcher.Dispatch(Event.FIRE_MISSILE,orientation);

    Listener

    缺点：Injection操作很慢

UFrame

和StrangeIOC设计思想很相似，包含很多文档与可视化编辑工具。

总结

可以选择性的用，UI部分用StrangeIOC等，动态加载用之前设计的池等。

管理准则

好架构等于好规则

编码准则

  • 使用C#
  • 命名使用一个有含义的命名
  • 起物体名称的时候尽量和场景对应起来，文件夹名称和代码名称尽量对应
  • 对warnings 和errors零容忍
  • 对动态开辟内存零容忍
  • 尽量避免GameObject.Find,GetComponent这种非常慢的操作
  • 美术资源严格管理
